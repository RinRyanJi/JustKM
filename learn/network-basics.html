⓪ 一句話總結

在學任何 API（包括 Alpaca、Shioaji、券商網站）之前，先把兩個核心概念搞清楚 — REST 與 WebSocket。

類型生活比喻特性Alpaca 用在哪
REST 📞 打電話叫外送
你打一次、店家回應一次、對話結束		 單次請求/回應、簡單、無狀態 查帳戶、下單、看持倉、看 K 線
WebSocket 📡 LINE 群組
雙方保持連線，誰有消息就 push 過去		 持續連線、雙向、伺服器主動推送 即時報價、即時成交、訂單更新推播
💡 最簡記法
  • REST = 「我去問，伺服器答，結束」→ 適合查資料、下單
  • WebSocket = 「我們保持通話，誰有消息就 push」→ 適合即時報價

① REST — 請問、回答、結束

REST 全名 Representational State Transfer，是現今網路最常用的 API 風格。 它的核心精神很簡單：你送一個「請求」給伺服器，伺服器回一個「回應」，然後對話就結束了。

想像你去餐廳：

  1. 你走到櫃台（送出請求）
  2. 你說「我要一份炒飯」（附帶資訊）
  3. 服務生說「好的，$100」（收到回應）
  4. 對話結束（連線關閉）

下次你想再點，就要重新走一次這個流程。伺服器不會記得你「上次」點過什麼。

② HTTP 4 大動詞（REST 的語言）

REST 用 4 個「動詞」告訴伺服器你要做什麼。這 4 個動詞是 HTTP 標準的，瀏覽器、APP、Python 程式都通用。

HTTP 動詞意思生活例子Alpaca 例子
GET查（不修改資料）看菜單查帳戶餘額 GET /v2/account
POST新增一筆下單點餐下單 POST /v2/orders
PATCH修改部分改訂單數量改單 PATCH /v2/orders/{id}
DELETE刪除取消訂單刪單 DELETE /v2/orders/{id}
📌 為什麼動詞不是中文？
HTTP 規範定義在 RFC 7231， 是全球通用的網路標準。無論你在台灣、日本、美國，用的動詞都一樣。

③ REST 流程圖（用 ASCII 圖看清楚）

下方這張圖展示一個 REST 請求的完整流程：

你（瀏覽器/程式） Alpaca 伺服器 │ │ │ ── GET /v2/account ──► │ (1) 你發出請求 │ │ (2) 伺服器查詢資料庫 │ ◄── JSON 資料 ────── │ (3) 回傳結果 │ │ │ ← 連線關閉 → │ (4) 結束 │ │

💻 實際範例（用 curl 問 Alpaca 帳戶）

bash 
curl -X GET \
  -H "APCA-API-KEY-ID: YOUR_KEY_ID" \
  -H "APCA-API-SECRET-KEY: YOUR_SECRET" \
  https://paper-api.alpaca.markets/v2/account

伺服器會回一段 JSON：

{
  "id": "904837e3-3b76-47ec-b432-046db621571b",
  "account_number": "PA1234567890",
  "status": "ACTIVE",
  "cash": "10000.00",
  "portfolio_value": "15000.00",
  "buying_power": "20000.00",
  ...
}

就這樣，一個請求、一個回應、結束。

④ HTTP Status Code（伺服器告訴你結果的編號）

伺服器回應的時候，除了 JSON 資料外，還會帶一個三位數的狀態碼。第一個數字代表大類別：

代碼範圍意思
2xx✅ 成功（你的請求 OK）
3xx🔄 重新導向（網址搬了）
4xx❌ 你端錯了（送錯格式、沒權限）
5xx💥 伺服器錯了（系統當機）

你會遇到的 9 個常見狀態碼

Code意思你該怎麼做
200✅ 成功拿資料用
201✅ 建立成功（POST 用）拿資料用
400❌ 你送出的格式錯了檢查 JSON、必填欄位
401❌ 沒帶 API Key檢查 Header
403❌ Key 錯或權限不足檢查 Key、餘額、PDT
404❌ 路徑錯了檢查 URL 拼字
422❌ 內容格式對但邏輯錯看 error message
429❌ 請求太頻繁睡一下再試（rate limit）
500💥 伺服器當機稍後再試或聯絡客服
⚠️ 菜鳥最容易搞混：404 不一定是「網頁不存在」， 在 API 中通常是「你打的 endpoint 路徑錯了」；403 不一定是「被禁止」， Alpaca 中常見是「帳戶餘額不足」或「PDT 限制」。

⑤ WebSocket — 一直保持連線，誰有消息誰就 push

WebSocket 是瀏覽器與伺服器之間的長連線。連線建立後，雙方隨時可以互相傳訊息，不用每次都重新「打招呼」。

想像你加入 LINE 群組：

  1. 你加入群組（建立連線）
  2. 群組成員開始聊天（雙向訊息）
  3. 有新訊息自動跳出來（伺服器主動推送）
  4. 你不必一直問「有新的嗎？」（不需要輪詢）
  5. 你退群才結束（手動關閉連線）

🎯 什麼時候該用 WebSocket？

  • ✅ 你需要即時資料（每秒數十筆的報價）
  • ✅ 你不想一直問伺服器「有新的嗎？」
  • ✅ 你想讓伺服器主動通知你（訂單成交、價格觸發）

反過來說，如果你只是想 5 秒鐘查一次 股價，REST 輪詢（每 5 秒送一次 GET）就夠了，WebSocket 反而太複雜。

⑥ WebSocket 流程圖

你（程式） Alpaca 伺服器 │ │ │ ── 請求升級成 WS ──► │ (1) 用 HTTP 發起升級請求 │ ◄── 101 Switching ─── │ (2) 伺服器同意升級 │ │ │ ← 連線保持 → │ (3) 持續連線（長時間） │ │ │ ── {"action":"auth"} ► │ (4) 你送認證 │ ── {"action":"subscribe"} ► │ (5) 你訂閱 BTC/USD 報價 │ │ │ ◄── {報價 1} ───── │ (6) 伺服器主動推播 │ ◄── {報價 2} ───── │ (7) 又一筆成交 │ ◄── {報價 3} ───── │ (8) 又有新報價 │ ◄── {報價 4} ───── │ │ │ │ ← 連線保持（直到你關） → │

🔧 連線範例（用裸 WebSocket）

python 
import websocket   # pip install websocket-client
import json

def on_open(ws):
    # 認證
    ws.send(json.dumps({
        'action': 'auth',
        'key': 'YOUR_KEY_ID',
        'secret': 'YOUR_SECRET_KEY'
    }))
    # 訂閱 BTC/USD 與 ETH/USD 的即時報價
    ws.send(json.dumps({
        'action': 'subscribe',
        'trades': ['BTC/USD'],
        'quotes': ['ETH/USD']
    }))

def on_message(ws, msg):
    arr = json.loads(msg)
    for item in arr:
        t = item.get('T')
        if t == 't':    # trade
            print(f"TRADE {item['S']} px={item['p']} qty={item['s']}")
        elif t == 'q':  # quote
            print(f"QUOTE {item['S']} bid={item['bp']} ask={item['ap']}")

ws = websocket.WebSocketApp(
    'wss://stream.data.alpaca.markets/v1beta2/crypto',
    on_open=on_open,
    on_message=on_message
)
ws.run_forever()   # 一直保持連線

注意：這個範例一旦啟動，就不會自動結束，因為它要持續接收推播。按 Ctrl+C 結束。

⑦ REST vs WebSocket 對比表

面向RESTWebSocket
連線模式短連線（問完就斷）長連線（一直保持）
發起方永遠是客戶端雙方都可以
通訊單位HTTP 請求/回應JSON 訊息（雙向）
即時性差（要輪詢）極佳（伺服器主動推）
資源消耗低（短連線）較高（持續佔用 socket）
複雜度簡單（curl 就能用）較複雜（需管理連線）
斷線處理無需處理需要重連邏輯
適合場景CRUD、報表、定期抓資料即時報價、聊天室、通知
Alpaca 應用下單、查持倉、看 K 線即時報價、訂單狀態推播
🤔 該用哪個？
• 想查資料、下單、看 K 線 → REST
• 想即時知道價格變動 → WebSocket
• 想被通知訂單成交 → WebSocket（trade_updates）
• 大多數情境 → REST 為主，WebSocket 為輔

⑧ 動手試：兩個都能跑的最簡範例

以下兩個 Python 範例完全不需要 Alpaca Key，用公開測試站即可跑通。先試試看，會更有感覺。

範例 A：REST（用 httpbin.org 模擬伺服器）

python 
import requests   # pip install requests

# GET 請求 — 像打電話問「現在幾點」
r = requests.get('https://httpbin.org/get')
print(f"狀態碼: {r.status_code}")   # 200
print(f"回應 JSON: {r.json()}")

# POST 請求 — 像打電話說「我要點一份炒飯」
r = requests.post('https://httpbin.org/post',
                  json={'symbol': 'BTC/USD', 'qty': 0.01})
print(f"伺服器說: {r.json()['json']}")   # 把你送的資料 echo 回來

預期輸出：

狀態碼: 200
回應 JSON: {'args': {}, 'headers': {...}, 'origin': '...', 'url': '...'}
伺服器說: {'symbol': 'BTC/USD', 'qty': 0.01}

範例 B：WebSocket（用 echo.websocket.org 測試）

python 
import websocket   # pip install websocket-client

# 建立連線
ws = websocket.create_connection('wss://echo.websocket.org')

# 你送一句話
ws.send('Hello, 伺服器!')

# 伺服器 echo 回來
reply = ws.recv()
print(f"伺服器說: {reply}")

# 關閉
ws.close()

預期輸出：

伺服器說: Hello, 伺服器!

這個範例中：你連上一個 echo 伺服器，送什麼它就回什麼。真實 Alpaca WebSocket 差別只在伺服器會主動推播行情，不只回應你的訊息。

💡 練習建議：
  1. 先跑範例 A（簡單），看到 200 跟 JSON 就成功了
  2. 再跑範例 B（稍難），看到 echo 回來就成功了
  3. 範例 B 如果一直沒收到回應，檢查防火牆（公司網路常擋 WS）
  4. 兩個都成功後，你已經準備好學任何 API 了 🎉
