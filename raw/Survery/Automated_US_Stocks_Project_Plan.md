# 自動化美股分析與自動交易 — 階段性專案計畫

本文檔為從入門到專業（production-grade）建立自動化美股分析與自動交易系統的分階段計畫，涵蓋每個階段的目標、成果、建議技術、資料來源、測試及時間估算。請將此檔案放置於 `/doc/Survery/`。

---

## 概要
- 目標：建立一個能自動分析美股、回測策略、並安全地進行自動下單的系統。
- 核心面向：資料、策略、回測、風控、執行/券商整合、部署與監控、合規。

---

## 階段總覽（從入門到專業）
1. 入門（理解與環境搭建） — 1–2 週
2. 初階（策略原型與回測） — 4–8 週
3. 中階（自動化與紙上交易） — 6–12 週
4. 進階（穩定化、監控、持續部署） — 8–16 週
5. 專業（低延遲執行、風控合規、資金管理與量產） — 3–6 個月

每個階段都包含：目標、產出（deliverables）、主要技術選項、資料來源、測試清單、預估工時。

---

## 第一階段：入門（理解與環境搭建）
- 目標：熟悉股票市場基礎、Python 資料科學生態、並建立能取得歷史與即時行情的環境。
- 產出：環境說明文件、範例程式（抓取歷史資料、繪圖）、簡單指標（MA、RSI）示例。
- 建議技術：Python, pandas, numpy, matplotlib/plotly, Jupyter Notebook
- 推薦資料來源（免費/付費）：Yahoo Finance（yfinance）, Alpha Vantage（免費配額有限）, IEX Cloud, Finnhub, Polygon（付費即時/整合）
- 重要任務：
  - 設置虛擬環境與程式庫（建議使用 `venv` 或 `conda`）
  - 撰寫資料拉取腳本並保存為 CSV
  - 實作並視覺化常見技術指標（SMA, EMA, RSI, MACD）
- 測試：資料完整性檢查、指標計算單元測試
- 時間估算：1–2 週

---

## 第二階段：初階（策略原型與回測）
- 目標：設計可回測的交易策略，能在歷史資料上進行模擬並評估績效。
- 產出：策略範例（例如均線交叉、動量策略）、回測報告（淨值曲線、夏普、最大回撤）、簡單交易日誌格式
- 建議技術／框架：backtrader, zipline（注意維護狀況）, vectorbt, bt, pandas
- 重要任務：
  - 定義策略介面（買/賣訊號函式）
  - 建立回測引擎或使用現成框架（處理手續費、滑點）
  - 評估常見績效指標（夏普、Sortino、勝率、回撤、CAR）
  - 引入參數搜尋（網格搜尋、隨機搜尋）與基礎交叉驗證（滾動回測）
- 資料注意：使用逐日（daily）與逐分鐘（1m/5m）資料測試不同粒度策略
- 測試：回測結果可重現性、邊界情況測試（極端波動、停牌）
- 時間估算：4–8 週

---

## 第三階段：中階（自動化與紙上交易）
- 目標：把策略從離線回測推到自動化程式，完成「紙上交易」（paper trading）並驗證實際執行情況。
- 產出：調度器（cron/airflow）、實時資料管道、紙上交易執行器、日誌與交易追蹤頁面
- 建議技術：Alpaca API（美股交易與紙上交易）、ib_insync（Interactive Brokers）、WebSocket 實時串流、FastAPI/Flask 做輕量服務
- 重要任務：
  - 建置資料管線（實時推送或定時拉取）
  - 策略容器化（Docker）並加入配置管理（環境變數或 Vault）
  - 與券商 API 做紙上交易連接與測試（含訂單簽章、資金限制）
  - 實作簡單風控（單筆風險、單日最大損失、單股持倉上限）
- 測試：紙上交易與回測產生的一致性、下單延遲量測、錯誤與重試流程驗證
- 時間估算：6–12 週

---

## 第四階段：進階（穩定化、監控、持續部署）
- 目標：提高系統可靠度與可觀察性，建立自動重啟、異常告警與持續部署流程。
- 產出：監控面板（Prometheus + Grafana）、日誌聚合（ELK 或 Loki）、CI/CD pipeline、回測/紙上交易綜合報表
- 建議技術：Docker Compose / Kubernetes（可選）、GitHub Actions / GitLab CI、Prometheus、Grafana、Sentry、Loki
- 重要任務：
  - 加入度量與追蹤（訂單成功率、延遲、策略績效）
  - 自動化測試（單元、整合、回測回歸測試）
  - 建立告警（Slack/Email）與事件處理流程
  - 定期策略再訓練與回測排程
- 測試：高可用情境模擬、資料中斷恢復測試、負載測試
- 時間估算：8–16 週

---

## 第五階段：專業（低延遲執行、風控合規、資金管理）
- 目標：達到能在真實資金下穩定營運的水準，符合券商與法規要求並擁有完善的風控與資金管理流程。
- 產出：現貨下單系統（production）、回溯與事件審計、風控庫、資金分配模組、法遵文件
- 建議技術：專業券商整合（Interactive Brokers 群組帳戶或券商 FIX 接入）、高可用部署（K8s）、監控與自動控停系統
- 合規與風控重點：
  - 熟悉美國監管（SEC、FINRA）、了解 Pattern Day Trader 規則、報稅與交易記錄保存要求
  - 建立 KYC/AML（若有對外資金流）與審計紀錄
  - 風控：回測一致性、實盤滑點適配、資金分配（Kelly/風險平價）、熔斷與停機機制
- 測試：小額真實資金驗證（逐步放大）、合規審查、壓力測試
- 時間估算：3–6 個月

## 技術棧建議（快速參考）
- 語言：Python（生態成熟）或 Rust/Go（低延遲核心模組）
- 資料處理：pandas, numpy, polars（高效）
- 回測：backtrader, vectorbt, zipline（注意社群活躍度）
- 機器學習：scikit-learn, xgboost, lightgbm, pytorch/torch, tensorflow
- 部署：Docker, Kubernetes, GitHub Actions
- 監控：Prometheus + Grafana, Sentry
- 券商 API：Alpaca, Interactive Brokers (IBKR), TD Ameritrade (注意 API 政策)

## 風控、合規與安全（基本清單）
- API Key 與私密資訊請使用安全儲存（Vault、Secrets Manager 或環境變數）
- 實盤前務必完成 paper trading 一段時間（數週到數月）
- 設置多層風控：單筆金額上限、當日最大虧損、最大持倉集中度、熔斷閾值
- 保存完整交易紀錄與審計日誌（含回測與實盤）以便稽核
- 咨詢法律顧問以確認是否需註冊或報備（視規模與服務對象而定）

---

## 測試與驗證清單
- 單元測試（指標、訊號計算）
- 回歸回測（確保改動不改變歷史結果）
- 紙上交易一致性測試（回測 vs 紙上）
- 故障與恢復測試（網路中斷、API 異常）
- 小量真實資金驗證（逐步上量）

---

## 建議時程樣板（範例 6 個月路線）
- 月 1: 完成第 1 階段（環境、資料管線、基本指標）
- 月 2–3: 完成第 2 階段（多策略回測、參數搜尋）
- 月 4: 完成第 3 階段（自動化、紙上交易）
- 月 5: 完成第 4 階段（監控、CI/CD、風控自動化）
- 月 6+: 進入第 5 階段（小量實盤驗證、合規準備、性能優化）
